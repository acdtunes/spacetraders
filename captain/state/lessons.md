L1 [seed] — Probes are cheap: keep 1 probe per 2-3 markets for price freshness before committing haulers.
L2 [seed] — No market prices without a ship physically at the waypoint; scouting is the only price-intel source.
L3 [seed] — Deploy ALL scouts to cover ALL markets; never leave a scout idle or scout only a subset.
L4 [seed] — Prefer solar probes as scouts: zero fuel = infinite runtime, they pay for themselves fast.
L5 [seed] — Always compute round-trip fuel (out + return + 10% margin); a stranded ship earns zero.
L6 [seed] — Contracts pay twice (accept + deliver) and are guaranteed; prefer over speculative mining/trading in bootstrap.
L8 [seed] — Source contract goods from EXPORT markets (cheapest); avoid IMPORT buys; mine only when no market option.
L9 [seed] — Buy at exporters, sell at importers — the most reliable arbitrage.
L13 [seed] — Declining $/h at steady yields = market saturation from selling the same export; diversify or cut volume.
L14 [seed] — Monitor both supply-chain sides: rising EXPORT prices mean an import shortage will break your arbitrage.
L15 [seed] — Routes aren't static; competition equilibrates prices — keep multiple routes, exit below-threshold margin.
L16 [seed] — Scale incrementally: validate profitability BEFORE buying ships; premature scaling depletes credits.
L18 [seed] — Fuel is a volatile exchange good; high-demand waypoints spike and can kill a route's margin.

L30 [d-9,d-10] — A socket hang (context deadline exceeded) can be spontaneous and may not self-recover in-session. Scope with ONE socket + ONE ledger probe (if ledger answers, the daemon isn't dead); don't hand-probe (token burn) — record, defer, it recovers between sessions. A multi-session blackout can be an out-of-band ops event, not a code bug.
L31 [d-8,d-10,d-86] — `workflow.finished success:true` is NOT proof of completion; cross-check — but the check cuts BOTH ways. The ledger QUERY (Postgres) can lag container events by HOURS, so an empty/stale ledger is NOT proof of a stall: the container log's explicit `Contract fulfillment transaction recorded in ledger` line is fresher ground truth than the ledger read (L28/L40). Trust the container log for whether a fulfillment happened; treat a "3h gap" in `ledger list` as lag until the log disproves it.
L32 [d-12,d-13] — `ship info` cargo/position/role can be a PHANTOM (stale cache) vs the server; the SERVER (delivery/sell API) is ground truth. A workflow failing on server cargo=0 is DETERMINISTIC — don't retry; clear the cache with `ship refresh` (L47).

L39 [d-19,d-20] — A merged daemon fix only takes effect after the daemon RESTARTS; a whole-cache desync self-clears on restart via GET /my/ships. Confirm a fix by EXERCISING (observed behavior), not status alone.
L40 [d-20,d-21,d-59] — A single 4203 fuel-crash (and transient API-error/404 bursts) is SELF-HEALING: the container auto-restarts, refuels, resumes; the coordinator re-spawns a fresh worker after a burst window. Don't intervene on the first crash — escalate only on a crash-LOOP (restart count climbing, no progress) or a ship idle >60min WITH cargo. Judge a route by NET ledger P&L, not fuel-line alarm.
L41 [d-22,d-23] — Contract payouts are LUMPY (one netted +155k, ~24× typical); derive $/h from several contracts, not one draw. Size the 50% guardrail against the biggest plausible cargo buy (~73k+), not the average.

L47 [d-34,d-39,d-87] — Phantom cache (cargo/position/role) RECURS after each contract; the in-band fix is `ship refresh --ship <sym>` (force GET /my/ships), now allowlisted (s32). It doesn't only break sell/deliver: a phantom FOREIGN-CARGO hold silently BENCHES a hauler from the coordinator's eligible pool (looks half-full → not claimable). A HAULER idle across sessions with stale partial cargo is a refresh candidate, not a lost ship (s81: TORWIND-3 cached 44/80 IRON_ORE, server 0/80). `ship refresh` ALONE re-arms it — the contract coordinator re-discovers idle light haulers IN-LOOP every contract-accept, so a phantom-cleared EXISTING-role hauler rejoins the eligible pool with NO container restart (s82/d-87: TORWIND-3 ran 3 contracts post-refresh). The L50 restart caveat is specific to ROLE-assignment visibility, not idle-hauler discovery.
L48 [d-35] — Contract cycle time is TRAVEL-dominated (~67%) and cadence is ENDOGENOUS; decompose ledger timestamps before calling $/h "supply-gated." Speed-blind "select closest" is a leak ONLY with 2+ eligible light haulers.

L56 [d-88] — To measure manufacturing/trading standalone $/h: `ledger list --type SELL_CARGO` isolates it CLEANLY — contracts emit CONTRACT_FULFILLED and NEVER SELL_CARGO, so gross trading revenue = sum(SELL_CARGO) with zero hand-pairing (only the BUY side stays muddied, L45). TWO gotchas: (a) the ledger displays in UTC-3 while container logs are UTC — align a sell to its task by the exact 3h offset (sell 21:53:20 local == task-complete 00:53:20 UTC); (b) a factory's FIRST collection dumps ACCUMULATED inventory, so the first COLLECT_SELL task's rate OVERSTATES steady state (task eebee24b: +229,740 rev / ~68,856 buy = ~+160k net in 4min — do not annualize one draw, L41).
L57 [d-90] — `construction start` creates an INERT pipeline (pipeline_type=CONSTRUCTION in manufacturing_pipelines, status PLANNING) but spawns NO coordinator — nothing executes it. The MANUFACTURING coordinator is the executor and it adopts pipelines ONLY at startup recovery (FindByStatus PLANNING/EXECUTING, NO type filter, NO re-poll in the main loop). Required order: `construction start` FIRST → THEN bounce `operations --manufacturing` (stop+start the coordinator) so recovery adopts the PLANNING construction pipeline ("Started recovered PLANNING pipeline <id>" confirms). Construction + manufacturing share ONE unified task queue and one `--max-workers` pool (no per-pipeline pinning) → a manufacturing backlog can STARVE gate tasks; verify the gate by MATERIAL progress climbing off 0.0%, not by pipeline status "EXECUTING". Corrects the s84 assumption of a separate dedicated construction coordinator/hauler (L55 was persistence-only).
L58 [d-90,s86] — "Not sold in-system" is a TIME-STAMPED observation, not standing truth: factory markets wake their export side once anyone (including YOUR OWN pipelines) feeds their imports. L59 [d-90] — Pipeline status EXECUTING proves PERSISTENCE+ADOPTION, not EXECUTION: the mfg coordinator happily adopts a CONSTRUCTION pipeline whose task type has NO activation path (every recurring PENDING→READY path is hard-filtered to FABRICATION/COLLECTION — supply_monitor/task_rescuer/pipeline_lifecycle) and NO registered executor (main.go:499-508) — tasks sit PENDING forever, silently. Queue priority is per-task and construction actually WINS it (DELIVER_TO_CONSTRUCTION 75 > COLLECT_SELL 50 > ACQUIRE_DELIVER 10), so "starvation" explanations must first prove the task ever became READY. Before betting a mission thread on a never-exercised subsystem: read the composition root for activation + executor registration, or demand first observable OUTPUT within one window.
L60 [d-91] — A merged code fix does NOT repair the state the buggy binary left behind: after N broken sessions, expect poisoned persisted rows (s87: an EXECUTING CONSTRUCTION pipeline whose tasks restart-recovery had reaped) to be a SECOND, independent blocker — and idempotent "resume" verbs turn that state into a permanent trap (StartOrResume returns any PLANNING/EXECUTING pipeline as-is; --depth ignored, no re-plan, nothing terminalizes a 0-task pipeline). When exercising a just-fixed subsystem, verify the DATA precondition (do the tasks/rows exist?) in the same breath as the code path, and put "tasks exist" BEFORE "output > 0" in the expectation chain.
